## 1. 프로젝트 개요 및 목적

**ticketFlow**는 콘서트/뮤지컬 티켓팅 플랫폼입니다. 사용자가 공연을 탐색하고, 예매 오픈 시각에 가상 대기실을 거쳐 좌석을 선택·결제하여 예매를 확정하는 흐름을 제공합니다.

| 항목 | 내용 |
|---|---|
| 제품 성격 | 학습/포트폴리오 프로젝트 (**실서비스 운영 목표 아님**) |
| 기간 | 3개월 (12주) MVP |
| 팀 | 백엔드 2명 (프론트엔드는 개발자 A가 AI 도구로 생성·유지보수) |
| 제품의 존재 이유 | **"동시 대량 접속 시 좌석 선점/중복 예매 방지"를 실제로 구현하고 수치로 증명하기 위한 도메인** |

티켓팅 도메인을 고른 이유는 기능이 화려해서가 아니라, **예매 오픈 순간에 트래픽이 한 점(하나의 회차, 하나의 좌석)으로 수렴하는 구조**가 동시성 문제를 가장 정직하게 드러내기 때문입니다. 이 프로젝트에서 공연 목록·상세·마이페이지 같은 기능은 **동시성 코어를 실제로 굴려보기 위한 최소한의 무대 장치**입니다.

## 2. 목표와 비목표 (Non-Goals)

### 2.1 목표

| # | 목표 | 검증 방법 |
|---|---|---|
| G1 | 동일 좌석에 대한 동시 예매 요청에서 **정확히 1건만 성공** | 동시성 통합 테스트(`ExecutorService`) + 11주차 부하 테스트 후 검증 SQL |
| G2 | 예매 오픈 순간의 트래픽을 **가상 대기실로 흡수**하고 순번대로 입장 허용 | 동시 진입 시 순번 중복 없음, 중복 진입 방지 동작 확인 |
| G3 | 결제 실패·이탈 상황에서 **좌석이 영구히 잠기지 않음** | PENDING 만료 스케줄러 + Redis TTL 만료 후 좌석 재판매 가능 확인 |
| G4 | 2인 병렬 개발에서 **상호 블로킹 없이** 12주 안에 완주 | Sync Point(SP0~SP7) 통과 (`docs/ROADMAP.md`) |
| G5 | 예매 확정 이벤트를 **비동기로 분리**하여 후속 처리(알림) 연동 | Kafka 발행 → consumer 수신 확인 |

### 2.2 비목표 (Non-Goals)

명시적으로 **만들지 않습니다.** 일정 압박 시 이 목록에 있는 항목을 다시 꺼내지 않기 위해 문서에 고정합니다.

| 제외 항목 | 이유 |
|---|---|
| 실제 PG 연동 | 결제는 Mock. `payment.pg_provider` 컬럼으로 교체 지점만 남겨둠 |
| 실명인증 / 소셜 로그인 | 인증은 자체 JWT 최소 구현이면 목표 검증에 충분 |
| 스포츠·전시 등 타 도메인 | 좌석 구조가 달라 스키마가 복잡해지고 동시성 검증에 기여하지 않음 |
| 실서비스급 배포·운영 (오토스케일링, 다중 인스턴스, 모니터링 스택) | 단일 서버 데모로 한정 |
| 실서비스급 관리자 권한관리 (감사 로그, 다단계 권한, 승인 워크플로) | 관리자 CRUD 자체는 **목표로 승격**(5.8절)했으나, `ADMIN` 단일 role 기반 최소 구현까지만. 권한체계 고도화는 동시성과 무관 |
| 좌석 등급별 할인·쿠폰·포인트 | 가격 계산 복잡도만 늘고 동시성과 무관 |
| Refresh Token | MVP 생략 가능으로 합의됨. 도입 시 Redis 저장 (`docs/ROADMAP.md` 1~2주차) |

> **v1.2에서 "관리자 백오피스(공연 등록/수정 UI)"를 비목표에서 제거하고 목표로 승격했습니다** (5.8절 FR-M1~M3). FR-B6 시드 데이터는 폐기하지 않고 **초기/부하 테스트 데이터를 멱등하게 대량 투입하는 수단으로 관리자 CRUD와 병존**합니다(수천 좌석을 화면으로 등록하는 것은 비현실적).

## 3. 대상 사용자

실서비스가 아니므로 시장 세그먼트가 아니라 **동작을 검증해야 하는 역할** 기준으로 정의합니다.

| 역할 | 정의 | 이 사람이 만드는 요구사항 |
|---|---|---|
| **일반 예매자** | 오픈 시각 이전부터 대기하다가 좌석을 잡으려는 사용자 | 대기 순번이 보여야 하고, 좌석 선택 후 결제까지 시간 여유가 있어야 하며, 실패 시 이유를 알아야 함 |
| **동시 접속 부하** (k6 가상 사용자) | 11주차 부하 테스트에서 동일 회차·동일 좌석에 몰리는 수천 개의 가상 사용자 | 락 경합 하에서도 중복 예매 0건, 응답이 무한 대기하지 않을 것 |
| **평가자 (면접관/리뷰어)** | 코드와 부하 리포트를 보고 설계 의도를 판단하는 사람 | 설계 결정의 **근거**가 문서로 남아 있을 것, 수치로 증명될 것 |
| **관리자/운영자** | 공연·회차·좌석·등급을 등록/수정/삭제해 예매 대상 데이터를 구성하는 사용자 | `ADMIN` 권한으로만 접근 가능한 CRUD, 이미 `SOLD`이거나 예매가 걸린 좌석·공연의 삭제 방지 |

> `users.role`의 `ADMIN`은 **v1.2부터 실제로 사용합니다** — 관리자 CRUD(5.8절)의 인가 기준입니다. 단 단일 role 기반 최소 구현이며 권한체계 고도화는 비목표입니다.

## 4. 핵심 기술 챌린지 — 좌석 선점 / 중복 예매 방지

이 프로젝트가 존재하는 이유이며, 아래 4겹의 방어선으로 해결합니다. **상세 구현 규칙과 함정은 [`CLAUDE.md`](../CLAUDE.md) "예매 확정 트랜잭션 흐름"과 [`docs/ROADMAP.md`](ROADMAP.md) 7~8주차를 참조하세요.**

| 층 | 장치 | 막는 것 |
|---|---|---|
| ① 트래픽 | Redis ZSET 가상 대기실 + 입장 토큰 | 오픈 순간의 요청 폭주가 좌석 API에 직접 닿는 것 |
| ② 임시 점유 | Redis 분산 락 `seat:lock:{session_seat_id}` (TTL, `SET NX EX` + Lua 해제) | 같은 좌석을 두 사람이 동시에 선택하는 것 |
| ③ 영구 정합성 | 부분 유니크 인덱스 `uq_booking_seat_active` (`WHERE status='ACTIVE'`) | **락이 뚫려도** DB가 물리적으로 두 번째 예매를 거부 |
| ④ 트랜잭션 배치 | T1(예매 생성·중복 검증) → 결제 → T2(확정) 2단계 분리 | "결제는 성공했는데 좌석은 뺏긴" 상태의 **구조적** 발생 |

### 요구사항으로 확정된 설계 결정

| 결정 | 요구사항으로서의 의미 |
|---|---|
| `session_seat`에 `HELD` 상태를 두지 않음 | 임시 점유는 Redis만, DB는 영구 상태(`AVAILABLE`/`SOLD`)만. Redis/DB 상태 불일치 버그를 원천 차단 |
| 좌석 락은 `RLock`이 아닌 `SET NX EX` + Lua 해제 | 좌석 점유는 **HTTP 요청을 넘어 유지**되므로 스레드 소유 기반 락은 사용 불가 (ROADMAP 함정 ③) |
| 다중 좌석 락은 **좌석 ID 정렬 순** 획득, 부분 실패 시 전량 해제 | 요청 간 데드락 방지 |
| 결제 호출은 **트랜잭션 밖** | 수 초짜리 외부 호출이 DB 커넥션을 점유하면 경합 구간에서 커넥션 풀 고갈 |
| Redis 락 해제는 **커밋 이후** | 해제 후 커밋하면 그 틈에 다른 사용자가 같은 좌석을 잡음 |
| TTL 3종의 값 고정 | `입장 토큰 15분 / 좌석 락 7분 / 결제 제한 5분` (B가 결정·관리, A는 화면 타이머를 이 값에 맞춤) |
| **TTL 잔여시간 불변식** (아래 참조) | 대소 관계만으로는 불충분 — 세 TTL은 **서로 다른 시점에 시작**하므로 각 전이 시점의 *잔여시간*을 검증해야 함 |

#### TTL 잔여시간 불변식 (P0 요구사항)

`15분 > 7분 > 5분`이라는 **대소 관계는 필요조건일 뿐 충분조건이 아닙니다.** 세 TTL의 기산점이 각각 다르기 때문에, 대소 관계를 지켜도 아래 반례가 성립합니다.

| 반례 | 결과 |
|---|---|
| 입장(t=0, 토큰 만료 t=15) → 배치도를 10분 둘러본 뒤 t=10에 선점(락 t=10~17) → t=16에 확정 요청 | **토큰은 이미 만료(t=15)**. 좌석은 잡혀 있는데 확정 불가 |
| 선점(t=0, 락 t=0~7) → t=6:50에 확정 시작 → 결제 제한 5분을 소진 | **결제 도중 t=7에 락 만료.** 타인이 같은 좌석 선점 성공 → 그 사용자는 409를 받지만 좌석은 아직 팔린 게 아님 (잘못된 에러 + 아무도 못 사는 구간) |

따라서 다음 두 검증을 **요구사항으로** 둡니다.

| ID | 검증 시점 | 불변식 | 위반 시 |
|---|---|---|---|
| INV-1 | 좌석 선점(FR-K1) 진입 시 | `입장 토큰 잔여 TTL ≥ 좌석 락 TTL` | 선점 거부 (또는 토큰 갱신 후 진행) |
| INV-2 | 예매 확정(FR-K3) 진입 시 | `좌석 락 잔여 TTL ≥ 결제 제한시간` | 확정 거부, 또는 **소유자 검증 후 Lua로 `PEXPIRE` 연장** |

> 대소 관계만 지키고 안심하지 마세요. `docs/study/02-developer-b-workflow.md`의 "TTL 3종을 여기서 확정하세요" 절에 같은 내용을 반영해 두었습니다. `CLAUDE.md`·`ROADMAP.md`는 아직 대소 관계만 서술하고 있습니다.

## 5. 기능 요구사항

우선순위 정의: **P0 = 없으면 핵심 목표(G1~G3) 검증 불가 / P1 = 데모 완성도에 필요 / P2 = 여유 시**

### 5.1 인증 (도메인 `user` — 담당 A)

| ID | 기능 | 우선순위 | 요구사항 |
|---|---|---|---|
| FR-A1 | 회원가입 | P0 | 이메일·비밀번호·이름 등록. 이메일 중복 시 거절(`uq_users_email` 위반도 동일 응답으로 변환). 비밀번호는 BCrypt 해시 저장 |
| FR-A2 | 로그인 | P0 | 이메일·비밀번호 검증 후 JWT Access Token 발급 (HS256, 만료 30분, `Authorization: Bearer`) |
| FR-A3 | 내 정보 조회 | P0 | 인증이 필요한 API가 실제로 보호되는지 확인하는 기준 API |
| FR-A4 | 인증 실패 응답 규격 | P0 | 401/403이 HTML이 아닌 **JSON**으로 반환 (프론트가 파싱 가능해야 함) |
| FR-A5 | Refresh Token | P2 | 비목표에 준함. 도입 시 Redis 저장 |
| FR-A6 | 내 정보 수정 | P1 | 이름 등 프로필 수정. `PATCH /api/users/me`. **이메일 변경 허용 여부는 U20 결정에 종속**(허용 시 `uq_users_email` 재검증 필요). 본인만 수정 가능(인가) |
| FR-A7 | 비밀번호 변경 | P1 | **현재 비밀번호 검증 후** 새 비밀번호를 BCrypt로 재해시. 새 비밀번호는 FR-A1과 동일한 입력 검증 규칙 재사용 |

### 5.2 공연 카탈로그 (도메인 `performance`/`venue`/`seat` — 담당 B)

| ID | 기능 | 우선순위 | 요구사항 |
|---|---|---|---|
| FR-B1 | 공연 목록 조회 | P0 | 페이징. 최신순 정렬. N+1 발생 금지(`fetch join`/`@BatchSize`) — **검증 가능한 형태로**: "목록 20건 조회 시 발생 쿼리 수 ≤ N"을 테스트로 고정 |
| FR-B2 | 공연 상세 + 회차 목록 | P0 | 회차별 `session_at` / `booking_open_at` / `booking_close_at` / 상태 노출 |
| FR-B3 | 좌석 잔여 요약 조회 | P0 | 등급·구역 단위 잔여 수. 전체 좌석 응답 없이도 화면이 뜨도록 분리 |
| FR-B4 | 좌석 배치도 조회 | P0 | 회차의 좌석 목록 + `AVAILABLE`/`SOLD` 상태 + 좌표(`pos_x`,`pos_y`). **최고 트래픽 API**이므로 응답 필드 최소화 필수 — **상한을 수치로**: "2000석 기준 gzip 후 X KB 이하" (X 미결정). ⚠️ PENDING 좌석은 `session_seat`가 아직 `AVAILABLE`이라 **배치도에 예매 가능으로 표시**됩니다 (U5가 2단계 확정이면 노출 창이 수 분 — 그 경우 "선택 불가" 표시 필요) |
| FR-B5 | 좌석 배치도 캐싱 | P2 | Redis 캐싱. **의도적 P2 강등** — `ROADMAP.md:303`은 "캐싱 1순위", `02-developer-b-workflow.md:456-463`은 3~4주차 과제로 격상해 두었으나, 부하 리포트 소재로서의 가치보다 P0 완주를 우선함 |
| FR-B6 | 시드 데이터 생성 | P0 | 공연장 1곳 + 좌석 500~2000석, 공연 2~3건, 회차 3~5개/공연, 등급 4종. **멱등**해야 하며 `booking_open_at`을 과거/수분 후/내일로 섞어 대기실 테스트를 가능하게 함 |

> FR-B6은 기능이 아니라 **전제 조건**입니다. `session_seat` 데이터가 없으면 좌석 선점 로직을 개발도 테스트도 할 수 없으므로 1~2주차 최우선입니다.

### 5.4 예매 (도메인 `booking` — 담당 B)

| ID | 기능 | 우선순위 | 요구사항 |

| FR-K7 | 내 예매 목록·상세 | P1 | 최신순 페이징(`idx_booking_user` 활용), 상세는 좌석·결제 정보 포함 |

> **FR-K4 — 예외를 뭉뚱그리지 말 것.** T1에는 `uq_booking_seat_active` 외에 `uq_booking_number`(`V1__init.sql:155`)도 걸려 있습니다. `DataIntegrityViolationException`을 통째로 "중복 예매"로 매핑하면, 예매번호 충돌이 사용자에게 **"이미 예매된 좌석"으로 오표시**되고 부하 리포트의 409 카운트까지 오염되어 S1·S2의 증거가 흐려집니다. (현재 `docs/study/02-developer-b-workflow.md:1052`가 이 광범위 catch 형태입니다 — 정정 대상)
>
> **FR-K5 — 만료의 의미는 U5에 종속됩니다.** 확정이 **단일 요청**이면 PENDING 수명은 밀리초이므로 이 스케줄러는 "T1 커밋 후 T2 전에 프로세스가 죽은 경우"만 잡는 크래시 복구 장치이고, `expires_at = now+5분`은 사실상 무의미합니다. **2단계 확정**이면 PRD 6.2의 "결제 중 이탈" 시나리오가 실재하고 이 스케줄러가 주 경로가 됩니다. → **U5를 먼저 결정하세요.**

### 5.5 결제 (도메인 `payment` — 담당 A)

| ID | 기능 | 우선순위 | 요구사항 |
|---|---|---|---|
| FR-P1 | Mock 결제 | P0 | 실 PG 대신 즉시 성공 응답. `transaction_id`를 UUID로 발급해 실 연동 시 구조가 유지되도록 함 |
| FR-P2 | 결제 실패 시나리오 | P1 | 실패를 인위적으로 발생시킬 수 있어야 T1 롤백 경로를 테스트할 수 있음. **선행 결정: 실패 이력을 `payment`에 `FAILED`로 남길 것인가** (아래 주석) |
| FR-P3 | 결제 재시도 허용 | P1 | **조건부 — FR-P2 결정에 종속.** 실패 이력을 적재하기로 하면 `uq_payment_booking`을 `WHERE status='SUCCESS'` 부분 유니크로 전환 (7절). 적재하지 않으면 **현재 스키마 그대로 문제없음** |

> **FR-P3의 근거를 정정합니다.** 초안은 "`UNIQUE(booking_id)` 때문에 실패 후 재시도 INSERT가 막힌다"를 전제로 부분 유니크 전환을 확정 사항으로 올렸으나, **확정된 흐름에서는 이 전제가 성립하지 않습니다.** 결제 실패 시 `payment` 행을 애초에 INSERT하지 않고(`docs/study/02-developer-b-workflow.md:1058-1063` — `cancelPending()`은 `booking`/`booking_seat`만 CANCELLED로 전환), `payment` INSERT는 T2에서 성공 시에만 일어납니다(같은 문서 1071행). 재시도는 booking이 CANCELLED된 뒤 **새 `booking_id`로** 시작되므로 유니크 제약에 걸리지 않습니다.
> → 순서를 바로잡습니다: **FR-P2(실패 이력 적재 여부)를 먼저 결정 → 적재하기로 할 때만 부분 유니크 전환.** `docs/study/01-developer-a-workflow.md:187-189`도 동일한 잘못된 전제를 공유하고 있어 함께 정정 대상입니다.

```
2. 공연 목록 → 공연 상세 → 회차 선택
        ↓
3. 대기실 진입 (POST 대기열)               → 순번 발급 (ZSET)
        ↓  (폴링으로 순번 조회 = heartbeat)
   [입장 허용] → 입장 토큰 발급 (TTL 15분)
        ↓
4. 좌석 배치도 조회 → 좌석 1~4석 선택
        ↓
5. 좌석 선점                               → 입장 토큰 검증 → Redis 락 (TTL 7분)
        ↓  (화면에 7분 카운트다운)
6. 예매 확정 요청
   ├ T1: booking(PENDING) + booking_seat(ACTIVE)   ← 중복 예매는 여기서 걸러짐
   ├ 결제 (Mock, 트랜잭션 밖, 제한 5분)
   └ T2: payment + booking(CONFIRMED) + session_seat(SOLD)
        ↓  커밋 후 락 해제 → Kafka 이벤트 발행
7. 예매 완료 → 마이페이지에서 예매 내역 확인
```

> 🔴 **위 6번 단계가 HTTP 요청 1회인지 2회인지가 U5(미결정)입니다.** 현재 그림은 단일 요청처럼 읽히지만, 그렇다면 "결제 제한 5분" 동안 사용자가 무엇을 하는 구간이 없습니다. 아래 6.2의 "결제 중 이탈" 시나리오도 단일 요청에서는 **발생할 수 없습니다.** 결정 후 이 절과 6.2를 함께 갱신하세요.

### 6.2 분기 시나리오

| 상황 | 시스템 동작 | 사용자에게 보이는 것 |
|---|---|---|
| 이미 다른 사람이 락을 잡은 좌석 선택 | 락 획득 실패 → 요청 거절 | "이미 선택된 좌석입니다" + 배치도 갱신 |
| 3석 중 1석만 락 실패 | 획득한 락 **전량 해제** 후 실패 | 부분 성공 없음 — 전체 재선택 |
| 락은 통과했으나 DB 중복 (락 뚫림) | T1에서 `DataIntegrityViolationException` → 409 | "이미 예매된 좌석입니다" |
| 결제 실패 | T1 롤백(`booking`·`booking_seat` → CANCELLED) | "결제에 실패했습니다" — **좌석은 이미 반환된 상태** |
| 선점 후 확정 없이 이탈 | 락이 TTL로 자동 만료 | 좌석이 다시 판매 가능해짐 |
| 결제 중 사용자 이탈 | PENDING을 스케줄러가 만료 처리 | ⚠️ **U5가 단일 요청 확정이면 이 시나리오는 성립하지 않습니다** (이탈할 시간이 없음) |
| 확정 진입 시 락 잔여 < 결제 제한시간 | INV-2 위반 → 거부 또는 소유자 검증 후 `PEXPIRE` 연장 | "시간이 만료되었습니다. 좌석을 다시 선택해 주세요" |
| 락 만료 후 타인이 같은 좌석 재선점 → 원소유자가 확정 시도 | 원소유자 T1이 `uq_booking_seat_active` 위반 → 409 | 두 사용자 모두 실패할 수 있는 구간 — **INV-2로 예방** |
| 확정 응답을 못 받고 사용자가 재요청 | FR-K9 멱등성 처리 | 자기 좌석에 "이미 예매됨"이 뜨지 않아야 함 |
| T1 커밋 후 T2 전에 프로세스 크래시 | `session_seat`는 `AVAILABLE`인 채 `booking_seat`만 ACTIVE로 남음 → 만료 스케줄러(FR-K5)가 회수 | 회수 전까지 배치도에는 판매 가능으로 보이나 확정 시 409 |
| 대기 중 브라우저 종료 | 폴링 중단 → heartbeat 미갱신 → 큐에서 제거 | 재진입 시 새 순번 |
| 대기 인원 0인 회차 | fast-path 즉시 통과 | 대기 화면을 스치듯 지나감 |
| 입장 토큰 없이 좌석 선점 직접 호출 | 토큰 검증 실패 → 거절 | 대기실 우회 불가 |

**정본은 [`src/main/resources/db/migration/V1__init.sql`](../src/main/resources/db/migration/V1__init.sql)입니다.** 스키마 소유권은 Flyway이며 `ddl-auto: validate`이므로 엔티티가 어긋나면 앱이 기동하지 않습니다.

총 10개 테이블 — `users / venue / venue_seat / performance / seat_grade / session / session_seat / booking / booking_seat / payment`

| 테이블 | 역할 | 요구사항 관점에서 중요한 점 |
|---|---|---|
| `users` | 회원 | 이메일 유니크, BCrypt 해시, `role`은 `USER`/`ADMIN`. **`ADMIN`은 v1.2부터 관리자 CRUD(5.8) 인가에 실제 사용** |
| `venue` / `venue_seat` | 공연장과 **물리 좌석** | 공연장에 고정되어 모든 공연이 재사용. `pos_x`/`pos_y`는 배치도 렌더링용 |
| `performance` | 공연 | **공연 1건 = 공연장 1곳.** 투어는 별도 공연으로 등록 |
| `seat_grade` | 등급·가격 | 공연에 종속(등급/가격은 공연마다 다름) |
| `session` | 회차 | `booking_open_at`이 **가상 대기실의 트리거 기준값**. 엔티티 클래스명은 `PerformanceSession` |
| `session_seat` | **회차별 판매 좌석 = 좌석 선점의 단위** | 상태는 `AVAILABLE`/`SOLD` 2개뿐. **`HELD` 없음** |
| `booking` | 주문 | 한 번에 2~4석 예매를 합산 결제·전체 취소하기 위해 좌석과 분리 |
| `booking_seat` | 주문에 포함된 개별 좌석 | **`uq_booking_seat_active`** 부분 유니크 인덱스가 중복 예매 최후 방어선 |
| `payment` | 결제 (Mock) | `booking_id` NOT NULL FK → **결제를 먼저 호출하는 순서는 물리적으로 불가능** |

| 항목 | 상태 | 이유 |
|---|---|---|
| `booking.expires_at TIMESTAMPTZ` | ✅ 확정 | PENDING 만료 스케줄러의 기준 컬럼이 현재 없음. **단, 컬럼의 *의미*(크래시 복구용 짧은 값 vs 사용자 결제 시간)는 U5 결정에 종속** |
| `notification` 테이블 신설 | ✅ 확정 | ERD에서 제외되어 있어 알림 이력 적재 불가 (FR-N2). **이벤트 ID 컬럼을 포함**해야 FR-N3(멱등성)이 성립 |
| `uq_payment_booking` 재설계 | 🟡 **조건부** | FR-P2(결제 실패 이력 적재 여부)를 먼저 결정. 적재할 때만 `WHERE status='SUCCESS'` 부분 유니크 전환이 정당화됨 — 5.5절 주석 참조 |
| 삭제 정책용 soft delete 컬럼 | 🟡 **조건부** | 관리자 CRUD(FR-M1~M3)의 삭제를 hard delete로 할지 상태전환/`deleted_at` 방식으로 할지에 종속 — **U19에서 결정**. 상태전환 방식이면 `performance`/`session`에 컬럼 추가 마이그레이션 필요 |

> **관리자 CRUD(5.8)는 기존 10개 테이블만으로 동작합니다** — `performance`/`session`/`session_seat`/`seat_grade`는 이미 존재하므로 **신규 테이블은 불필요**. 위 soft delete 컬럼만 U19 결정에 따라 조건부로 추가될 수 있습니다.

> **인증 필요 여부는 위 표가 초안**이며, 카탈로그 조회를 비로그인에 열지 여부는 SP2에서 확정합니다.

> **`/api/admin/**`는 인증(✅)에 더해 `ADMIN` role을 요구합니다** — `SecurityConfig`에서 `hasRole('ADMIN')`으로 강제(9.1.1). 관리자 화면(FR-F9)이 이 계약에 연동되며, ADMIN 계정 부여 방식은 **U17**로 남깁니다. 삭제 시 참조 무결성 처리(U19)는 5.8절 참조.

## 9. 비기능 요구사항

### 9.1 정합성 (최우선 — 성능보다 앞섬)

| 항목 | 기준 |
|---|---|
| 중복 예매 | **0건.** 단, 아래 "검증 쿼리" 주석 참조 — 단순 `count(*)>1` 쿼리는 증거가 되지 못합니다 |
| 좌석 영구 잠김 | 없음. 모든 실패 경로에서 Redis TTL 또는 PENDING 만료 스케줄러가 좌석을 반환 |
| 대기열 순번 | 동일 사용자 재진입 시 순번 유지. 폴링 끊긴 유령을 포함해 순번이 과대 표시되는 오차 범위를 리포트에 명시 |
| 남의 락 해제 | 발생 불가. Lua로 "값이 내 소유자 값일 때만 DEL" (같은 사용자의 다른 탭 간 오삭제는 hold 단위 랜덤 토큰으로 해결 — **도입 여부 미결정**) |
| 남의 예매 취소 | 발생 불가. 소유자 확인 필수 |
| 에러 코드 분포 | 중복 시도가 **500이 아니라 409로 변환**되었는지 (FR-K4 판별이 실제로 동작한 증거) |

> **⚠️ 검증 쿼리를 바꿔야 합니다.** 초안의
> `SELECT session_seat_id, count(*) FROM booking_seat WHERE status='ACTIVE' GROUP BY 1 HAVING count(*)>1`
> 은 **항진명제**입니다. `uq_booking_seat_active`(`V1__init.sql:196-198`)가 존재하는 한 DB가 구조적으로 0행을 보장하므로, 이 결과는 "애플리케이션 코드가 옳다"가 아니라 "인덱스가 존재한다"의 증거일 뿐입니다.
>
> 실제로 의미 있는 것은 **상태 간 불일치**입니다. 부하 테스트 후 아래 5종을 모두 확인하세요.

| # | 확인 대상 | 뜻하는 것 |
|---|---|---|
| V1 | `session_seat.status='SOLD'`인데 대응하는 ACTIVE `booking_seat`가 없음 | 좌석 증발 |
| V2 | `booking.status='CONFIRMED'`인데 `session_seat`가 `AVAILABLE` | T2 부분 실패 |
| V3 | `booking.status='CANCELLED'`인데 `booking_seat`가 ACTIVE | 보상 누락 |
| V4 | `booking.status='CONFIRMED'`인데 `payment` 행 없음 (또는 그 역) | T2 정합성 깨짐 |
| V5 | 만료 시각이 지난 PENDING `booking` 잔존 수 | 만료 스케줄러 미동작 |

### 9.1.1 보안

| 항목 | 요구사항 |
|---|---|

### 9.2.1 가용성 — Redis 장애 시 동작 (정의 필요)

`docker-compose.yml:11-13`의 redis 서비스에는 **볼륨이 없습니다**(postgres만 `pgdata`). 컨테이너를 재생성하면 대기열 ZSET·입장 토큰·좌석 락이 **전부 소실**됩니다.

| 소실 대상 | 결과 |
|---|---|
| 입장 토큰 | 검증이 전부 실패 → 대기실을 통과했던 사용자가 좌석 선점 불가 (fail-closed, 안전하지만 데모 중단) |
| 좌석 락 | 동시 선점 폭증 → 409 폭증. **정합성 자체는 DB 제약이 지킴** |

논리적으로는 4절 ③층이 방어하지만, **가용성 정책이 비기능 요구사항에 없습니다.** fail-closed를 확정할지, 데모 중 Redis 재기동을 금지할지, 복구 절차를 둘지 정하세요.

## 11. 성공 기준 (포트폴리오 관점 — 무엇을 증명해야 하는가)

이 프로젝트는 "동작하는 티켓팅 사이트"가 아니라 **"동시성 문제를 이해하고 해결했다는 증거"**를 만드는 것이 목적입니다.

| # | 증명해야 할 것 | 산출물 (있어야 통과) |
|---|---|---|
| S1 | 동일 좌석 동시 요청에서 **1건만 성공**한다 | 동시성 통합 테스트 코드 + 통과 로그 |
| S2 | **락이 뚫려도** DB가 막는다 (다층 방어) | 제약 이름 기반 판별로 409를 내는 코드 경로(FR-K4) + **9.1의 V1~V5 상태 불일치 쿼리 전부 0행** + 409/500 에러 코드 분포 |
| S3 | 락 없는 버전과 **비교 수치**가 있다 | FR-K11 토글로 측정한 비교 그래프 — **DB 무결성 예외 수 / 409 비율 / p95·p99 / 커넥션 사용량** (중복 예매 0건은 락이 없어도 성립하므로 비교 지표가 될 수 없음) |
| S4 | 대기열이 트래픽을 실제로 **흡수**한다 | 대기실 유무 대비 — **오픈 직후 초당 요청 수 / 에러율 / p99** |
| S5 | 설계 결정에 **근거**가 있다 | 왜 `HELD`를 안 뒀는지 / 왜 `RLock`이 아닌지 / 왜 T1·T2를 나눴는지를 문서로 설명 가능 |
| S6 | 2인이 **막히지 않고** 병렬로 개발했다 | 인터페이스 선(先)합의 → 병렬 구현 이력이 남은 PR 로그 + **"상대 대기로 중단된 일수"를 SP마다 기록** (PR 로그만으로는 블로킹 부재의 증거가 되지 못함) |
| S7 | 전체 흐름이 **화면으로 데모**된다 | 로그인 → 대기 → 좌석 선택 → 결제 → 확정까지 끊김 없이 동작하는 데모 |
| S8 | 아키텍처를 설명할 수 있다 | 백엔드/아키텍처 다이어그램 + 부하 리포트 (12주차) |
